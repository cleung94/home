Understanding LOD and DAX: Two Ways to Think About Data

A beginner-friendly explanation of Tableau LOD expressions, Power BI DAX measures, how they are different, and how I used both in a physical security systems dashboard project.

April 2026 15 min read

When I started building my Physical Security Systems Health KPI project, I thought the hardest part would be making the dashboard look clean. I quickly learned that the real challenge was not the dashboard design. It was the logic behind the numbers.

The dataset looked simple at first. I had devices from two systems, CCure and Genetec, and each device could either be healthy or have one or more issues. But once I started cleaning the data, I ran into a very common analytics problem:

One device can appear on multiple rows, but it should not always be counted multiple times.

That single idea forced me to learn two important concepts: LOD expressions in Tableau and DAX measures in Power BI.

Both LOD and DAX helped me answer the same core question:

How do I count devices correctly when the data has multiple rows per device?

The Problem: Row-Level Data vs Device-Level Questions

Before talking about LOD and DAX, it helps to understand the problem they were solving.

In my project, each device could have multiple reasons for failure. For example, one camera might be:

  • Offline
  • Out of Standard
  • Using the wrong firmware

After cleaning the data, that one device could appear on three rows: 

Device Reference Number | Failure Type
CAM00001                | Offline
CAM00001                | Out of Standard
CAM00001                | Incorrect Firmware Version

This is good for counting issues, because I can count each failure separately. But it creates a problem when I want to count devices.

If I use a regular row count, Power BI or Tableau might count CAM00001 three times. But for system health, that device should only count as one failed device.

The Big Lesson: Always Know the Grain of Your Data

The word grain means the level of detail in your dataset.

In simple terms:

  • Device-level grain: one row per device
  • Issue-level grain: one row per issue

My cleaned dataset became issue-level because each row represented one failure type. That was useful for discrepancy reporting, but not enough for device health reporting.

That is where LOD and DAX came in.

What Is LOD in Tableau?

LOD stands for Level of Detail.

In Tableau, an LOD expression lets you tell Tableau:

Calculate this at this specific level, even if the chart is showing something else.

In my case, I wanted Tableau to calculate device health at the device level, even though the dataset had multiple rows per device.

The basic idea is: 

{ FIXED [Device Reference Number] : calculation }

This tells Tableau:

For each Device Reference Number, calculate one result.

LOD Example From My Project

This is the type of LOD logic I used to classify a device as healthy or failed: 

{ FIXED [Device Reference Number] :
    IF MAX(
        IF [Failure Type] = "No Reason for Failure"
        THEN 0 ELSE 1 END
    ) = 1
    THEN "Failed"
    ELSE "Healthy"
END
}

At first, this looks intimidating. But it becomes easier when broken down.

Breaking Down the LOD Expression

1. FIXED Device Reference Number

{ FIXED [Device Reference Number] : ... }

This tells Tableau to evaluate the calculation once per device. It does not matter how many rows that device has.

Think of it like saying:

“Group everything by device first, then run the logic.”

2. Convert failure rows into numbers

IF [Failure Type] = "No Reason for Failure"
THEN 0 ELSE 1 END

This part changes the failure text into a simple flag.

  • No Reason for Failure = 0
  • Any actual failure = 1

So if a device has three rows, Tableau might see: 

Offline                  = 1
Out of Standard          = 1
Incorrect Firmware       = 1

Or for a healthy device: 

No Reason for Failure    = 0

3. Use MAX to check if any issue exists

MAX(...)

The MAX function looks at all rows for the device.

If any row is 1, the max is 1. That means the device has at least one failure.

If all rows are 0, the max is 0. That means the device is healthy.

4. Return Failed or Healthy

IF MAX(...) = 1
THEN "Failed"
ELSE "Healthy"
END

This converts the result into a label that can be used in the dashboard.

So the final logic becomes:

  • If the device has at least one issue → Failed
  • If the device has no issues → Healthy

A Simple Way to Remember LOD

Here is the easiest way I remember LOD:

LOD tells Tableau what level to think at.

If the chart is messy, but I need the calculation to happen at the device level, I use LOD.

When Would You Use LOD?

LOD is useful when you need to calculate something at a different level than what the chart is showing.

Good use cases include:

  • Counting one result per device
  • Calculating customer-level metrics from order-level data
  • Finding the first or last date per user
  • Calculating totals that should not change when the chart changes
  • Creating fixed pass/fail classifications

In my project, I used LOD because I needed device-level logic while working with issue-level rows.

What Is DAX in Power BI?

DAX stands for Data Analysis Expressions.

DAX is the formula language used in Power BI to create calculated columns, measures, and tables.

If LOD tells Tableau what level to calculate at, DAX tells Power BI:

Calculate this based on the current filter context.

Filter context means the filters currently affecting the visual. This can include slicers, chart selections, page filters, or the fields inside the visual itself.

DAX Example From My Project

Here is the DAX measure I used for configuration compliance: 

Config Fail Devices =
SUMX(
    VALUES(DeviceAudit[Device Reference Number]),
    VAR HasConfigIssue =
        CALCULATE(
            COUNTROWS(DeviceAudit),
            KEEPFILTERS(
                FILTER(
                    DeviceAudit,
                    CONTAINSSTRING(DeviceAudit[Failure Type], "Configured Incorrectly")
                )
            )
        )
    RETURN IF(HasConfigIssue > 0, 1, 0)
)

This measure looks more complex than the Tableau LOD expression, but it is doing a similar job.

It checks each unique device and asks:

Does this device have a configuration issue?

If yes, count it as 1 failed device. If no, count it as 0.

Breaking Down the DAX Measure

1. VALUES creates a list of unique devices

VALUES(DeviceAudit[Device Reference Number])

This creates a temporary list of unique devices based on the current filters.

If the visual is filtered to CCure only, then the list contains CCure devices. If it is filtered to Genetec, it contains Genetec devices.

2. SUMX loops through each device

SUMX(...)

SUMX is an iterator. That means it goes row by row through a table and calculates something for each row.

In this case, it loops through each unique device.

Think of it like Power BI asking:

“For each device in this list, should I count it as failed?”

3. VAR stores a temporary result

VAR HasConfigIssue = ...

VAR creates a temporary variable. This makes the measure easier to read.

Here, the variable stores whether the current device has a configuration issue.

4. CALCULATE changes the context

CALCULATE(...)

CALCULATE is one of the most important functions in DAX. It evaluates an expression under specific filter conditions.

In this measure, CALCULATE checks the rows for the current device and applies the filter for configuration issues.

5. FILTER searches for the issue

FILTER(
    DeviceAudit,
    CONTAINSSTRING(DeviceAudit[Failure Type], "Configured Incorrectly")
)

This searches the Failure Type field and checks whether it contains the text Configured Incorrectly.

If it finds that text, the device has a configuration issue.

6. KEEPFILTERS preserves the visual context

KEEPFILTERS(...)

KEEPFILTERS prevents the measure from ignoring existing filters.

This mattered because I wanted the measure to respect filters like:

  • System = CCure
  • System = Genetec
  • Specific site or device type filters

7. Return 1 or 0

RETURN IF(HasConfigIssue > 0, 1, 0)

If the device has at least one configuration issue, return 1. If not, return 0.

SUMX then adds up all the 1s to produce the total number of failed devices.

A Simple Way to Remember DAX

Here is how I think about DAX:

DAX asks, “Given the filters right now, what should this number be?”

DAX is powerful because it reacts to slicers, filters, and visuals dynamically.

When Would You Use DAX?

DAX is useful when you need reusable measures that respond to dashboard filters.

Good use cases include:

  • KPI cards
  • Pass/fail measures
  • Percentages
  • Year-over-year calculations
  • Running totals
  • Dynamic compliance checks

In my project, I used DAX to create reusable measures for:

  • Total devices
  • Passed devices
  • Failed devices
  • Percent pass
  • Percent fail
  • Total discrepancies
  • Configuration compliance
  • Naming compliance
  • Firmware compliance
  • IP compliance
  • Device standard compliance

LOD vs DAX: The Main Difference

LOD and DAX can solve similar problems, but they think differently.

A simple way to compare Tableau LOD and Power BI DAX: LOD fixes the calculation level, while DAX responds to filter context.

Tableau LOD Thinks Like This

Tableau LOD says: 

Calculate this at the device level.

It is very direct. You tell Tableau the level of detail you want.

Power BI DAX Thinks Like This

Power BI DAX says: 

Based on the current filters, loop through the devices and calculate the result.

It is more context-driven. The answer changes depending on filters and visuals.

Comparison Table

Concept Tableau LOD Power BI DAX
Main purpose Control level of aggregation Create reusable calculations and measures
Best for Fixed calculations at a specific grain Dynamic calculations based on filters
Common use Per-device or per-customer calculations KPI cards, ratios, compliance measures
How it thinks “Calculate at this level” “Calculate based on the current context”
Project example Classify each device as healthy or failed Count failed devices by compliance category

Why This Was Hard at First

The hardest part was not writing the formula. The hardest part was understanding what I was actually trying to count.

At different times, I needed different answers:

  • How many unique devices exist?
  • How many devices failed?
  • How many total issues exist?
  • How many devices failed a specific compliance category?

Those are not the same question.

Once I understood that, the formulas started to make more sense.

Device Counts vs Issue Counts

This was probably the most important lesson of the whole project.

If I am measuring system health, I should count devices: 

COUNTD([Device Reference Number])

or in Power BI: 

Total Devices =
DISTINCTCOUNT(DeviceAudit[Device Reference Number])

But if I am measuring total discrepancies, I should count issue rows: 

COUNT([Failure Type])

or in Power BI: 

Total Discrepancies =
CALCULATE(
    COUNT(DeviceAudit[Failure Type]),
    DeviceAudit[Failure Type] <> "No Reason for Failure"
)

One device can have three issues. That means:

  • Device count = 1
  • Issue count = 3

That difference matters a lot in dashboard reporting.

How This Helped My Dashboard

Once I understood LOD and DAX, I was able to build dashboards that answered different levels of questions.

High-level questions

  • What is the overall pass rate?
  • What is the overall fail rate?
  • How many total devices are in the environment?

Operational questions

  • Which failure types are most common?
  • Which system has more discrepancies?
  • Which compliance category has the most failures?

Technical questions

  • Are devices configured correctly?
  • Are devices in standard?
  • Are cameras using IPv4?
  • Are firmware versions correct?

LOD and DAX helped me move from raw data to meaningful answers.

Common Mistakes I Ran Into

1. Counting rows when I meant to count devices

This caused inflated numbers because one device could appear multiple times.

2. Counting devices when I meant to count issues

This made discrepancy totals too low because multiple issues on the same device were collapsed.

3. Forgetting filter context in Power BI

Some DAX measures initially ignored the system filter, which made CCure and Genetec numbers look wrong.

4. Assuming Tableau and Power BI think the same way

They can produce the same answer, but the calculation logic is written differently.

Final Takeaway

Learning LOD and DAX helped me understand that dashboards are not just about visuals. The real work is making sure the numbers mean what they are supposed to mean.

LOD taught me how to control the level of detail in Tableau. DAX taught me how to create reusable measures in Power BI that respond to filters and context.

The biggest lesson was this:

Before building a dashboard, understand what each row represents and what each metric is supposed to count.

Once that clicked, the formulas became less intimidating. They became tools for answering better questions.

Key Takeaways

  • LOD expressions in Tableau control the level at which a calculation happens.
  • DAX measures in Power BI calculate values dynamically based on filter context.
  • LOD is useful when you need fixed calculations, such as one result per device.
  • DAX is useful when you need reusable KPI and compliance measures.
  • Device counts and issue counts are different and should be handled separately.
  • The most important part of dashboard logic is understanding the grain of the data.
