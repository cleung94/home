Overview

Turning raw system exports into an audit story

The goal of this project was to create a realistic data workflow that mirrors the kind of review a security operations, systems administration, or infrastructure analytics team might perform after exporting records from an access control platform and a video management platform.

Using a synthetic dataset with approximately 140 records, I designed the project to answer operational questions around device health, naming standards, lifecycle compliance, network configuration, and firmware consistency. The result is a single case study that shows data collection logic, QA thinking, technical analysis, and dashboard communication in one place.

This project was intentionally built as a portfolio piece for data analyst and data scientist style roles while still reflecting my domain background in physical security systems.

Business Questions

The audit questions this project answers

Core audit checks

  1. Are all devices online and configured correctly?
  2. Do programming names follow the correct naming conventions?
  3. Are all products currently used within our systems in-standard?
  4. Are all cameras in Genetec currently using IPv6?
  5. Are all cameras in Genetec online?
  6. Are all cameras in Genetec with the correct firmware version?

Naming convention used

Both systems follow the same format:

SITE CODE (LOCATION) DESCRIPTION [DEVICE REFERENCE NUMBER]

Example door:
SITE01 (FIRST FLOOR) LOBBY DOOR [DR00001]

Example camera:
SITE01 (FIRST FLOOR) LOBBY CAMERA [CAM00001]

Doors Readers Control Panels Cameras

Workflow

From data preparation to insight generation

1. Mock data generation

I created a synthetic export that resembles the type of fields a team might review in CCure and Genetec, including site code, location, device type, online status, firmware version, IPv6 usage, naming compliance, and lifecycle standard dates.

2. SQL audit logic

I used SQL to answer the business questions directly from the export tables. This shows how an analyst can turn operational checks into repeatable queries for validation, reporting, and site-level issue summaries.

3. Python summarization

I used Python and pandas to combine the audit flags into a concise summary table, then created a visualization that ranks issue counts by question to quickly highlight the biggest problem areas.

4. Dashboard storytelling

Finally, I translated the same findings into BI dashboard layouts for Power BI and Tableau to show how the same dataset can support multiple reporting styles depending on the audience.

Code

Example SQL and Python used in the project

SQL example

SELECT
  'Products In Standard' AS audit_question,
  CASE WHEN SUM(issue_count) = 0 THEN 'YES' ELSE 'NO' END AS all_clear,
  SUM(issue_count) AS issue_count
FROM (
  SELECT SUM(Issue_Standard_Flag) AS issue_count FROM ccure_export
  UNION ALL
  SELECT SUM(Issue_Standard_Flag) AS issue_count FROM genetec_export
);

This query checks whether active products are still within standard based on the expected end-of-standard date.

Python example

answers = pd.DataFrame([
    {
        "Question": "Are all Genetec cameras online?",
        "All Clear": "NO" if genetec["Issue_Offline_Flag"].sum() > 0 else "YES",
        "Issue Count": int(genetec["Issue_Offline_Flag"].sum())
    }
])

plot_df = answers.sort_values("Issue Count")
plt.figure(figsize=(10, 5))

Results

What the synthetic audit revealed

22 devices flagged

A combined total of 22 devices were not both online and correctly configured, indicating where health and setup checks would need follow-up.

15 naming exceptions

Several device records intentionally failed the expected programming format, making it possible to demonstrate naming governance checks.

42 lifecycle issues

The largest issue category involved products that had passed their expected end-of-standard date, creating a clear reporting story for lifecycle planning.

Genetec-specific gaps

Within the camera data, the project surfaced 10 IPv6 gaps, 8 offline cameras, and 7 firmware mismatches.
