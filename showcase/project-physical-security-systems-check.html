Physical Security Systems Health KPI

This project analyzes CCure and Genetec device data to evaluate the health of a physical security environment. I built the project from planning through data cleanup, transformation, analysis, and dashboard reporting in both Tableau and Power BI.

The goal was to turn raw audit-style system exports into a clear health and compliance report that answers whether devices are online, configured correctly, within company standards, using the correct firmware, and properly configured for IP standards.

Audit Questions

What I wanted the analysis to answer

System health questions

  1. What is the health of our physical security systems on a global scale?
  2. In both systems, are all devices online?
  3. In both systems, are all devices configured correctly?

Compliance questions

  1. In both systems, are all devices within company standards?
  2. In both systems, are all devices with the correct firmware?
  3. In Genetec, are cameras currently configured as IPv4?

Process

How I built the audit from raw exports to dashboards

1

Defined the audit questions

I started by deciding what the dashboard needed to answer before building any visuals. This helped me avoid creating charts without a clear purpose.

The final audit questions focused on global health, online status, configuration accuracy, company standards, firmware compliance, and Genetec IPv4 configuration.

2

Reviewed the CCure and Genetec exports

I reviewed the source data to understand which fields could be used across both systems. CCure represented access control devices, while Genetec represented camera and video surveillance devices.

I preserved the System field throughout the project so every record could still be traced back to either CCure or Genetec after cleanup.

3

Created validation logic

I created logic to identify whether each device passed or failed specific checks. These checks included online status, configuration, naming convention, company standard, firmware, and IP compliance.

This step turned the dataset from a raw export into an audit-ready dataset.

4

Reshaped the Reason for Failure field

The Reason for Failure column contained multiple issues in one cell, which made it difficult to count each issue individually. 

Before:
Device A | Offline; Out of Standard; Firmware Issue

After:
Device A | Offline
Device A | Out of Standard
Device A | Firmware Issue

I used Power Query to split the column by delimiter and then unpivot the results into separate rows.

5

Separated device counts from discrepancy counts

After unpivoting, one device could appear multiple times if it had multiple issues. Because of that, I separated the logic into two types of reporting.

For device health, I used distinct device counts. For discrepancy reporting, I counted issue rows.

6

Built the Tableau and Power BI dashboards

I built the final outputs in both Tableau and Power BI using the same cleaned dataset. Tableau helped with visual exploration and dashboard layout, while Power BI helped structure reusable KPI and compliance measures.

Challenges

Pain points and how I solved them

Multiple failures in one field

The Reason for Failure column contained multiple values in a single cell, which made it impossible to properly count individual issue types.

Solution: I used Power Query to split the column by delimiter and unpivot the results into rows so each failure could be counted separately.

Tableau did not support the transformation needed

Tableau did not provide the flexibility needed to reshape the dataset the way I needed.

Solution: I performed all transformation work in Power Query before importing the cleaned dataset into Tableau.

Devices appearing multiple times

After unpivoting, devices with multiple issues appeared multiple times, which could lead to inflated counts.

Solution: I separated device-level metrics (DISTINCTCOUNT) from issue-level metrics (COUNT).

Header row counted as data

During Power BI import, the header row was mistakenly counted as a device, causing an incorrect total.

Solution: I corrected the Power Query step by setting the first row as headers and verified the device count returned to 130.

Automation

Generating the audit workbook with Python

Why I used Python for this step

Before building the Tableau and Power BI dashboards, I needed a repeatable way to create the audit workbook. The goal was not only to combine CCure and Genetec data, but to generate a structured Excel file that could support analysis, cleanup, and dashboard reporting.

Python helped automate the workbook creation process so I did not have to manually rebuild the same Excel structure every time the source data changed. This made the project more consistent and easier to reproduce.

What the Python script produced

  • A combined audit workbook containing both CCure and Genetec records
  • A merged tab used as the main dashboard dataset
  • Consistent column names across both systems
  • A clear System field to separate CCure from Genetec
  • A structured Excel output that could be cleaned further with Power Query

Python script

import pandas as pd

# Load the source system exports
ccure = pd.read_excel("ccure_export.xlsx")
genetec = pd.read_excel("genetec_export.xlsx")

# Add a system identifier before combining the data
ccure["System"] = "CCure"
genetec["System"] = "Genetec"

# Combine both systems into one audit dataset
merged = pd.concat([ccure, genetec], ignore_index=True)

# Keep only the fields needed for audit analysis
merged = merged[[
    "System",
    "Site Code",
    "Location",
    "Device Type",
    "Device Reference Number",
    "Programming Name",
    "Reason for Failure"
]]

# Make healthy devices explicit
merged["Reason for Failure"] = merged["Reason for Failure"].fillna("No Reason for Failure")

# Export the cleaned structure into an Excel workbook
with pd.ExcelWriter("access_control_audit_results.xlsx", engine="openpyxl") as writer:
    ccure.to_excel(writer, sheet_name="CCure Export", index=False)
    genetec.to_excel(writer, sheet_name="Genetec Export", index=False)
    merged.to_excel(writer, sheet_name="Merged", index=False)

print("Audit workbook generated successfully.")

Key point: adding the System field

Before combining the data, I added a System column to both exports. This was important because once the records were merged together, I still needed to know whether each device came from CCure or Genetec.

This decision allowed the dashboards to compare both systems side by side instead of treating all devices as one generic dataset. 

ccure["System"] = "CCure"
genetec["System"] = "Genetec"

Key point: merging both exports

I used pd.concat() to stack the CCure and Genetec records into one merged dataset. This is similar to an append or union operation.

This was the correct approach because both systems represented device records. I did not want to join the tables side by side; I wanted one combined list of devices. 

merged = pd.concat([ccure, genetec], ignore_index=True)

Key point: selecting only useful audit fields

After merging the data, I narrowed the dataset to the columns needed for the audit. This helped remove unnecessary fields and kept the final workbook focused.

The selected fields supported the main dashboard questions: which system the device belonged to, where it was located, what type of device it was, how it was programmed, and why it failed. 

merged = merged[[
    "System",
    "Site Code",
    "Location",
    "Device Type",
    "Device Reference Number",
    "Programming Name",
    "Reason for Failure"
]]

Key point: making healthy devices explicit

Blank failure values can be confusing in analysis because they may look like missing data instead of healthy devices. To avoid that, I replaced blank failure reasons with No Reason for Failure.

This made it easier to separate healthy devices from failed devices later in Tableau and Power BI. 

merged["Reason for Failure"] =
    merged["Reason for Failure"].fillna("No Reason for Failure")

Key point: exporting multiple workbook tabs

I kept the original CCure and Genetec exports as separate tabs while also creating a merged tab. This preserved the source data while giving the dashboards a clean combined dataset to use.

This structure made the workbook easier to review because the original system exports stayed available as reference tabs. 

with pd.ExcelWriter("access_control_audit_results.xlsx", engine="openpyxl") as writer:
    ccure.to_excel(writer, sheet_name="CCure Export", index=False)
    genetec.to_excel(writer, sheet_name="Genetec Export", index=False)
    merged.to_excel(writer, sheet_name="Merged", index=False)

Why this mattered later

The Python-generated workbook became the foundation for the rest of the project. Once the merged tab existed, I could continue shaping the data with Power Query, split the failure reasons, unpivot the issue columns, and load the cleaned result into Tableau and Power BI.

This step made the workflow more reliable because the dashboard was not built from scattered files. It was built from a structured audit workbook.
