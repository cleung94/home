Physical Security Systems Health KPI

This project analyzes CCure and Genetec device data to evaluate the health of a physical security environment. I built the project from planning through data cleanup, transformation, analysis, and dashboard reporting in both Tableau and Power BI.

The goal was to turn raw audit-style system exports into a clear health and compliance report that answers whether devices are online, configured correctly, within company standards, using the correct firmware, and properly configured for IP standards.

Audit Questions

What I wanted the analysis to answer

System health questions

  1. What is the health of our physical security systems on a global scale?
  2. In both systems, are all devices online?
  3. In both systems, are all devices configured correctly?

Compliance questions

  1. In both systems, are all devices within company standards?
  2. In both systems, are all devices with the correct firmware?
  3. In Genetec, are cameras currently configured as IPv4?

Process

How I built the audit from raw exports to dashboards

1

Defined the audit questions

I started by deciding what the dashboard needed to answer before building any visuals. This helped me avoid creating charts without a clear purpose.

The final audit questions focused on global health, online status, configuration accuracy, company standards, firmware compliance, and Genetec IPv4 configuration.

I reviewed the source data to understand which fields could be used across both systems. CCure represented access control devices, while Genetec represented camera and video surveillance devices.

3

Created validation logic

I created logic to identify whether each device passed or failed specific checks. These checks included online status, configuration, naming convention, company standard, firmware, and IP compliance.

This step turned the dataset from a raw export into an audit-ready dataset.

4

The Reason for Failure column contained multiple issues in one cell, which made it difficult to count each issue individually.

Challenges

Pain points and how I solved them

Multiple failures in one field

The Reason for Failure column contained multiple values in a single cell, which made it impossible to properly count individual issue types.

Solution: I used Power Query to split the column by delimiter and unpivot the results into rows so each failure could be counted separately.

Tableau did not support the transformation needed

Tableau did not provide the flexibility needed to reshape the dataset the way I needed.

Solution: I performed all transformation work in Power Query before importing the cleaned dataset into Tableau.

Devices appearing multiple times

After unpivoting, devices with multiple issues appeared multiple times, which could lead to inflated counts.

Solution: I separated device-level metrics (DISTINCTCOUNT) from issue-level metrics (COUNT).

Header row counted as data

During Power BI import, the header row was mistakenly counted as a device, causing an incorrect total.

Solution: I corrected the Power Query step by setting the first row as headers and verified the device count returned to 130.

Automation

Generating the audit workbook with Python

Why I used Python

I used Python to automate the validation of physical security systems and eliminate the need for manual review across CCure and Genetec exports. Instead of checking each device individually, the script evaluates system health at scale and produces a repeatable audit output.

This step ensured consistency across all validation checks while also creating a structured dataset that could later be transformed for Tableau and Power BI dashboards.

What the script produced

The script generates a new Excel workbook called access_control_audit_results.xlsx with five structured tabs:

  • CCure Export: Raw CCure data preserved for reference.
  • Genetec Export: Raw Genetec data preserved for reference.
  • Device Standards: Reference sheet for lifecycle and firmware validation.
  • Audit Report: Executive-style audit summary with methodology and results.
  • Failed Devices: Remediation list with detailed failure reasons.

How the Python analysis works

The script uses pandas for data manipulation and openpyxl for Excel formatting. Pandas allows efficient row-level validation, filtering, and transformation, while openpyxl ensures the final workbook is formatted for readability and stakeholder use.

The data is loaded from CSV exports and a standards reference sheet. These are converted into lookup dictionaries to validate device models, lifecycle status, and firmware compliance.

A fixed project date is used to ensure consistent lifecycle validation results, making the audit reproducible regardless of when the script is executed.

Validation logic and flags

The script creates working copies of the CCure and Genetec datasets and adds validation flags for each type of audit check. Each flag represents whether a device fails a specific condition.

  • Online and configuration status
  • Programming name format validation
  • Programming name field matching
  • Device standard compliance
  • IPv6 usage (Genetec)
  • Firmware compliance (Genetec)

These flags use binary values (0 = pass, 1 = fail), allowing the script to aggregate results easily and calculate issue counts.

Why specific functions were used

.copy() was used to preserve the original datasets and avoid modifying raw exports.

.apply() was used for row-level evaluations, especially when validating programming names against multiple fields.

lambda functions were used for concise conditional checks, such as identifying out-of-standard devices and incorrect firmware versions.

Regular expressions (re) were used to enforce strict naming conventions for programming names.

Audit report generation

The script compiles audit results into a structured DataFrame that answers key validation questions. These results are written into the Audit Report tab, along with methodology and summary insights.

This allows stakeholders to quickly determine system health without needing to interpret raw data.

Failed device identification

A helper function builds a readable Reason for Failure string for each device by combining all failed checks into a single field. 

Offline; Out of Standard; Incorrect Firmware Version

This output provides a remediation-ready list that can be used by engineers to quickly identify and resolve issues.

After generating the Excel workbook, I used Power Query to reshape the data for visualization.

The Reason for Failure column initially contained multiple failure types in a single cell. This format is useful for human readability but not ideal for dashboards.

To prepare the data, I performed the following steps in Power Query:

Tableau Build

How I built the Tableau dashboard and logic

1

Connected to the cleaned dataset

I connected Tableau to the cleaned Excel workbook and used the merged audit tab as the primary data source. This dataset already contained normalized failure types and a consistent structure across CCure and Genetec.

This ensured that Tableau did not need to perform any additional data reshaping and could focus only on analysis and visualization.

2

Validated the device count

The first validation step was confirming that Tableau recognized 130 unique devices. Since devices could appear multiple times after unpivoting failure reasons, I avoided using simple row counts. 

COUNTD([Device Reference Number])

Why this mattered: This became the foundation for all device-level metrics and ensured that the dashboard was not inflated by duplicate rows.

3

Created device-level health classification (LOD expression)

Because each device could have multiple rows (one per failure type), I needed a way to evaluate the device as a whole. I used a Level of Detail (LOD) expression to determine whether each device had any failure. 

{ FIXED [Device Reference Number] :
    IF MAX(
        IF [Failure Type] = "No Reason for Failure"
        THEN 0 ELSE 1 END
    ) = 1
    THEN "Failed"
    ELSE "Healthy"
END
}

How this works:

  • FIXED [Device Reference Number] ensures the calculation runs once per device
  • IF [Failure Type] = "No Reason for Failure" identifies healthy rows
  • MAX() checks if any row for that device contains a failure
  • If any failure exists → device is marked as Failed

Why this mattered: This prevented a device with multiple issues from being counted multiple times in the pass/fail summary.

4

Built KPI metrics

Using the LOD classification, I created KPI values for total devices, passed devices, failed devices, and percentages. 

COUNTD([Device Reference Number])
COUNTD(
IF [Device Status] = "Failed"
THEN [Device Reference Number]
END)

Why this mattered: These KPIs gave a high-level overview of system health before showing detailed breakdowns.

5

Separated device counts from discrepancy counts

I intentionally separated device-level metrics from discrepancy-level metrics. Device counts use COUNTD, while discrepancies use row counts. 

COUNT([Failure Type])

Why this mattered: One device can have multiple issues, so discrepancy reporting needed to reflect total findings, not just affected devices.

6

Built discrepancy breakdown chart

I created a bar chart using Failure Type as the dimension and issue counts as the measure, with System as the color grouping.

This showed:

  • Which failure types occurred most often
  • Whether those failures came from CCure or Genetec

Why this mattered: It provided insight into what types of issues were driving the audit results.

7

Created compliance logic using conditional calculations

For each compliance category, I created calculated fields that flagged whether a device had a specific type of issue. 

IF CONTAINS([Failure Type], "Configured Incorrectly")
THEN "Fail"
ELSE "Pass"
END

I repeated this pattern for:

  • Configuration compliance
  • Naming convention compliance
  • Device standard compliance
  • IP address compliance
  • Firmware compliance

Why this mattered: Each compliance chart answered one of the audit questions directly.

8

Built system comparison views

I used the System field (CCure vs Genetec) in every major chart. This allowed me to compare performance across both systems instead of treating the data as a single group.

Why this mattered: It showed whether issues were isolated to one platform or shared across both.

9

Designed the dashboard layout

I structured the dashboard in layers:

  • KPI summary (top)
  • System distribution
  • Discrepancy breakdown
  • Compliance checks

Why this mattered: This layout allows the viewer to understand the overall health first, then drill into specific issues.

10

Final formatting and validation

I applied consistent colors across the dashboard:

  • Green = Pass
  • Red = Fail
  • Distinct colors for CCure and Genetec

I also verified that:

  • Device counts always totaled 130
  • Discrepancy counts reflected issue rows
  • All visuals aligned with the audit questions

Why this mattered: Formatting improved readability, while validation ensured the dashboard was accurate.

Power BI Build

How I built the Power BI dashboard and measures

1

Loaded the merged audit workbook

I imported the cleaned Excel workbook into Power BI and used the merged audit table as the main dataset. I renamed the table to DeviceAudit so the DAX measures were easier to read, reuse, and troubleshoot.

This helped keep the Power BI model organized because all dashboard measures referenced one main audit table.

2

Validated the imported data

Before creating visuals, I checked that Power BI was reading the dataset correctly. One issue I ran into was that the header row was briefly counted as a device, which caused the total device count to show 131 instead of 130.

I corrected this in Power Query by confirming the first row was promoted as headers and then verified that the final unique device count matched the expected 130 devices.

3

Created the Total Devices measure

The first measure I created was Total Devices. This measure counts unique device reference numbers instead of counting rows.

This was important because after splitting and unpivoting the failure reasons, a single unhealthy device could appear on multiple rows. A normal row count would overstate the number of devices. 

Total Devices =
DISTINCTCOUNT(DeviceAudit[Device Reference Number])

Why this measure mattered: It became the denominator for the pass rate, fail rate, and compliance calculations. It also acted as a validation check to make sure all dashboard sections still tied back to 130 unique devices.

4

Created Failed Devices and Passed Devices

Next, I created measures to separate devices that had at least one issue from devices that had no reason for failure.

The Failed Devices measure counts unique devices where the Failure Type is not equal to No Reason for Failure. 

Failed Devices =
CALCULATE(
    DISTINCTCOUNT(DeviceAudit[Device Reference Number]),
    DeviceAudit[Failure Type] <> "No Reason for Failure"
)

Then I calculated passed devices by subtracting failed devices from total devices. 

Passed Devices =
[Total Devices] - [Failed Devices]

Why this measure mattered: It created the main health split for the dashboard. Instead of manually filtering healthy and unhealthy devices, Power BI could calculate the counts dynamically based on the current filter context.

5

Created % Pass and % Fail

After creating the pass and fail counts, I created percentage measures so the dashboard could show overall system health at a glance. 

% Pass =
DIVIDE([Passed Devices], [Total Devices])

% Fail =
DIVIDE([Failed Devices], [Total Devices])

I used DIVIDE() instead of the normal division operator because it handles divide-by-zero situations more safely.

Why this measure mattered: Counts are useful, but percentages make the result easier to understand quickly. A viewer can immediately see the overall health ratio instead of calculating it mentally.

6

Created Total Discrepancies

I created a separate measure for discrepancies because discrepancies are different from devices. A device can have more than one issue, so discrepancy reporting should count issue rows instead of unique devices. 

Total Discrepancies =
CALCULATE(
    COUNT(DeviceAudit[Failure Type]),
    DeviceAudit[Failure Type] <> "No Reason for Failure"
)

Why this measure mattered: This measure shows the total number of findings, not the number of impacted devices. That distinction helped separate system health from remediation workload.

For example, one device with three issues should count as one failed device but three discrepancies.

7

Created Failure Count for the discrepancy chart

When I built the discrepancy breakdown chart, I needed a measure that counted failure rows by failure type while excluding healthy devices. 

Failure Count =
CALCULATE(
    COUNT(DeviceAudit[Failure Type]),
    DeviceAudit[Failure Type] <> "No Reason for Failure"
)

This measure was used in the bar chart with Failure Type as the axis and System as the legend.

Why this measure mattered: It allowed the dashboard to show which types of problems appeared most often and whether they were coming from CCure, Genetec, or both.

8

Created device-level compliance measures

The compliance visuals required device-level logic. I could not simply count rows because one device might have multiple failure rows.

For each compliance category, I used DAX to check whether each unique device had at least one matching failure type. If it did, the device failed that category. If it did not, the device passed. 

Config Fail Devices =
SUMX(
    VALUES(DeviceAudit[Device Reference Number]),
    VAR HasConfigIssue =
        CALCULATE(
            COUNTROWS(DeviceAudit),
            KEEPFILTERS(
                FILTER(
                    DeviceAudit,
                    CONTAINSSTRING(DeviceAudit[Failure Type], "Configured Incorrectly")
                )
            )
        )
    RETURN IF(HasConfigIssue > 0, 1, 0)
)

How this works:

  • VALUES(DeviceAudit[Device Reference Number]) creates a list of unique devices in the current filter context.
  • SUMX() loops through each device one at a time.
  • FILTER() searches that device's rows for the compliance issue.
  • CONTAINSSTRING() checks whether the Failure Type contains the target issue text.
  • KEEPFILTERS() preserves the current chart filters, such as CCure or Genetec.
  • If the device has at least one matching issue, the measure returns 1 for that device.

Why this measure mattered: It prevented overcounting and allowed each compliance chart to show accurate device-level pass/fail results.

9

Created compliance pass measures

After calculating the number of devices that failed a specific compliance check, I created pass measures by subtracting the fail count from total devices. 

Config Pass Devices =
[Total Devices] - [Config Fail Devices]

I reused this same pattern for the other compliance categories:

  • Naming Compliance: checks for Programming Name issues
  • Device Standard Compliance: checks for Out of Standard issues
  • IP Compliance: checks for IPv4 issues
  • Firmware Compliance: checks for Firmware issues

Why this measure mattered: It made each compliance visual easy to build. Each chart only needed the system field on the axis and the pass/fail measures as the values.

10

Repeated the compliance pattern for each audit category

Once the configuration compliance measure worked, I reused the same pattern for the other categories by changing the search condition. 

-- Naming
CONTAINSSTRING(DeviceAudit[Failure Type], "Programming Name")

-- Device Standard
DeviceAudit[Failure Type] = "Out of Standard"

-- IP Address
CONTAINSSTRING(DeviceAudit[Failure Type], "IPv4")

-- Firmware
CONTAINSSTRING(DeviceAudit[Failure Type], "Firmware")

Why this measure pattern mattered: Reusing the same structure made the report consistent. Every compliance chart followed the same logic, which made the results easier to validate and explain.

11

Built the Power BI visuals

After creating the measures, I built the report layout using KPI cards, a system distribution chart, a discrepancy breakdown chart, and clustered column charts for compliance.

  • KPI cards: Total Devices, % Pass, % Fail, and Total Discrepancies
  • System distribution: total devices by CCure and Genetec
  • Discrepancy breakdown: issue count by failure type and system
  • Compliance charts: pass/fail device counts by system

Why this mattered: The visuals were designed to move from high-level health to detailed issue categories. This made the dashboard easier to read because the viewer could understand the summary first, then investigate the details.

12

Formatted and validated the final dashboard

Finally, I formatted the visuals so the report was easier to scan. I used green for pass, red for fail, and separate system colors for CCure and Genetec.

I also validated that the device-level visuals tied back to 130 total devices, while discrepancy visuals correctly reflected issue counts instead of unique device counts.

Why this mattered: The final dashboard needed to be accurate and readable. Formatting helped with readability, but validation ensured the numbers were trustworthy.

Tool Comparison

Comparing Tableau and Power BI for the same audit dataset

Why I built the dashboard in both tools

I wanted to understand how the same cleaned dataset would behave in both Tableau and Power BI. Since the data questions were the same, the comparison helped me focus on how each tool handles calculations, layout, interactivity, and dashboard design.

Building both versions also helped me better understand the difference between Tableau’s visual-first workflow and Power BI’s measure-driven workflow.

Main difference I noticed

Tableau felt faster for visual exploration and layout testing. Power BI required more setup through DAX measures, but once the measures were created, the dashboard logic felt more reusable and structured.

The biggest difference was how each tool handled device-level logic. Tableau used LOD expressions, while Power BI used DAX measures with SUMX, VALUES, and filter context.

Tableau calculation approach

In Tableau, the most important concept was using LOD expressions to control the level of detail. Since the dataset had multiple rows per device after unpivoting, I used a FIXED calculation to classify each device once. 

{ FIXED [Device Reference Number] :
    IF MAX(
        IF [Failure Type] = "No Reason for Failure"
        THEN 0 ELSE 1 END
    ) = 1
    THEN "Failed"
    ELSE "Healthy"
END
}

This made Tableau better at answering the question: Is this device healthy or failed?

Power BI calculation approach

In Power BI, the same device-level logic was handled through DAX. The measure loops through each unique device, checks whether the device has a matching failure type, and returns a pass/fail count. 

Config Fail Devices =
SUMX(
    VALUES(DeviceAudit[Device Reference Number]),
    VAR HasConfigIssue =
        CALCULATE(
            COUNTROWS(DeviceAudit),
            KEEPFILTERS(
                FILTER(
                    DeviceAudit,
                    CONTAINSSTRING(DeviceAudit[Failure Type], "Configured Incorrectly")
                )
            )
        )
    RETURN IF(HasConfigIssue > 0, 1, 0)
)

This made Power BI better at creating reusable compliance measures across multiple visuals.

Final takeaway

The biggest lesson from building both versions was that dashboard accuracy depends on understanding the grain of the data. A row-level issue count and a device-level health count are not the same thing.

Tableau and Power BI both solved the problem, but each required a different approach. Tableau relied on LOD expressions, while Power BI relied on DAX measures and filter context. Building both versions helped me better understand how to translate the same business logic across different BI tools.

Reflection

What I learned and how I would improve the next iteration

What worked well

This project successfully transformed raw system exports into a structured audit dataset and dashboard. One of the most important outcomes was separating device-level health from issue-level discrepancies. This allowed the dashboard to clearly answer both high-level health questions and detailed operational questions.

The workflow I built — planning, cleaning, transforming, and then visualizing — created a repeatable process that could be reused for similar datasets in the future.

Key lessons learned

  • Data structure matters more than visualization
  • Row-level data and entity-level data must be handled differently
  • Upfront data cleanup simplifies dashboard logic significantly
  • Each audit question should map directly to a calculation or visual

Limitations of the current version

The current version of the dashboard provides a snapshot view of system health at a single point in time. While this is useful for identifying current issues, it does not show how system health changes over time.

Because the dataset is based on a single export, it cannot answer questions such as:

  • Are systems improving or degrading over time?
  • Which issues are recurring vs newly introduced?
  • Are remediation efforts reducing discrepancies?

Next iteration: adding trend analysis

If this project were extended, the next step would be to introduce a time-based data model to support trend analysis. This would allow the dashboard to move from a static audit report to a monitoring tool.

Instead of working with a single dataset, I would collect and store multiple audit snapshots over time.

  • Add a snapshot date column to each dataset
  • Store historical audit results instead of overwriting them
  • Track device status changes between snapshots
  • Measure how discrepancy counts change over time

Trend metrics I would introduce

  • Device health trend (% pass over time)
  • Total discrepancies trend
  • New vs resolved issues
  • Compliance trend by category
  • System-level comparison over time (CCure vs Genetec)

These metrics would help answer whether the environment is improving or declining instead of only showing the current state.

Visualization improvements

To support trend analysis, I would add:

  • Line charts showing pass rate over time
  • Stacked area charts for issue distribution
  • Monthly or weekly breakdowns of discrepancies
  • Filters for date ranges and system comparison

This would allow users to move from a static report to an interactive monitoring dashboard.

Data pipeline improvements

To support ongoing analysis, I would improve the data pipeline by automating data collection and storage.

  • Automate export ingestion from CCure and Genetec
  • Store historical data in a database instead of flat Excel files
  • Schedule regular data refreshes
  • Create a structured data model for reporting

This would reduce manual work and ensure the dashboard always reflects the most recent data.

Final reflection

This project started as a static audit exercise but highlighted how important data modeling and transformation are in building meaningful dashboards. The biggest improvement for the next iteration would be shifting from a one-time analysis to a continuous monitoring solution.

Adding time-based tracking, automation, and trend analysis would make the dashboard more valuable for ongoing decision-making instead of only point-in-time reporting.
